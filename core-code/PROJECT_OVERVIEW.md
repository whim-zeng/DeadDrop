# DeadDrop（死信箱）— 项目总览

> 基于地理位置的"数字纸条"匿名社区  
> 架构模式：Orchestrator-Worker（complex-project skill）  
> 硬件核心：ESP32-S3

---

## 一、产品愿景

人们在特定地点有强烈的表达欲望（餐厅吐槽、景点感悟、校园告白、医院候诊焦虑），但现有社交平台是"人的网络"而非"地点的网络"。

**DeadDrop 让用户在当前位置"放下"一条匿名纸条 → 只有物理到达该位置半径50米内的人才能"捡起"阅读 → 纸条有生命周期（被阅读次数决定寿命）→ 支持回复形成匿名对话链 → 与 AI "土地公公"对话了解此地故事。**

## 三、模块详情

### 软件域（Software Domain）

| 模块 ID | 模块名称 | 核心职责 | 技术栈 |
|---------|----------|----------|--------|
| SW-01 | 移动端应用层 | 用户主入口：留/读纸条、地图浏览、AGENT聊天、社交匹配 | React Native + Expo + Geolocation API |
| SW-02 | 后端服务与空间数据 | 纸条CRUD、空间查询（50m半径）、生命周期管理、用户系统 | Supabase (PostgreSQL + PostGIS) + Edge Functions |
| SW-03 | AI 服务层 | 内容审核（OpenAI Moderation）、地点AGENT生成、对话系统 | OpenAI API + 自建 Agent 框架 |
| SW-04 | 实时通信与社交 | 匿名回复、双方同意后的长期聊天、消息推送 | Supabase Realtime + Push Notification |
| SW-05 | 运营与数据可视化 | 纸条全景地图（热力图）、运营后台、品牌投放管理 | React Admin / 独立 Dashboard + Mapbox/Leaflet |

### 硬件域（Hardware Domain）

| 模块 ID | 模块名称 | 核心职责 | 平台/技术 |
|---------|----------|----------|-----------|
| HW-01 | ESP32-S3 物理终端设备 | 部署在实体场所的"数字信箱"，无手机也可交互 | ESP32-S3 (Xtensa LX7, 2 cores, 512KB SRAM) |
| HW-02 | 固件与嵌入式系统 | 设备启动、WiFi/BLE 连接、HTTP/MQTT 通信、状态机 | ESP-IDF (FreeRTOS) + Arduino framework 备选 |
| HW-03 | 外设集成与驱动 | 触摸屏/电子墨水屏、按键、蜂鸣器、LED 状态指示、GPS 模块 | LVGL (GUI) + 各类传感器驱动 |
| HW-04 | 设备管理与 OTA | 远程配置、固件空中升级、设备注册与认证、心跳上报 | ESP HTTPS OTA + 设备影子 |
| HW-05 | 电源与功耗管理 | 电池供电优化、低功耗模式、太阳能充电（户外场景） | FreeRTOS 电源管理 + 电池监测 |

## 四、核心交互流程

```
【用户通过手机 App 交互】
用户到达地点 → GPS 定位 → 后端空间查询(50m) → 显示附近纸条 → 留/读/回复

【用户通过 ESP32-S3 终端交互】
用户走近设备 → 触摸屏幕/按键 → WiFi 连接后端 → 显示该地点纸条 → 留/读
→ LED/蜂鸣器反馈 → 设备进入低功耗等待

【地点 AGENT 生成流程】
地点纸条积累 → 触发 AI 分析 → 生成"土地公公"角色设定 → 存入向量数据库
→ 用户提问 → RAG 检索相关纸条 → 生成个性化回复

【社交匹配流程】
A 回复 B 的纸条 → B 收到通知 → B 同意建立连接 → 双方进入长期聊天频道
```

## 五、商业化场景映射

| 场景 | 软件支持 | 硬件部署 |
|------|----------|----------|
| 品牌地理营销（门店优惠券纸条）| SW-05 品牌投放后台 | HW-01 门店内终端 |
| 旅游景区合作（景点故事频道）| SW-03 地点AGENT + SW-05 付费频道 | HW-01 景区打卡点终端 |
| 校园告白/活动 | SW-04 匿名回复 + 聊天 | HW-01 食堂/图书馆终端 |
| 城市寻宝游戏 | SW-02 空间数据 + SW-05 活动管理 | HW-01 多个线索点终端 |

## 六、Epic 依赖关系图

```
Phase 1（基础设施）
  ├── SW-02 后端服务与空间数据  ←── 最先启动
  └── HW-02 固件基础 + HW-05 电源管理

Phase 2（核心功能）
  ├── SW-01 移动端应用层  ←── 依赖 SW-02
  ├── SW-03 AI 服务层     ←── 依赖 SW-02
  └── HW-03 外设集成      ←── 依赖 HW-02

Phase 3（集成与扩展）
  ├── SW-04 实时通信与社交  ←── 依赖 SW-01 + SW-02
  ├── SW-05 运营后台      ←── 依赖 SW-02 + SW-03
  ├── HW-01 完整终端组装   ←── 依赖 HW-02 + HW-03 + HW-05
  └── HW-04 设备管理 OTA   ←── 依赖 HW-01

Phase 4（商业化）
  └── 品牌对接 + 付费频道 + 大规模硬件部署
```
